# YuKiKo 项目深度总览与维护手册（发布前终版）

> 文档定位：这是一份给“准备发布 + 持续维护”场景的工程手册。
> 目标不是只讲概念，而是把项目当前的架构、运行链路、风险点、Prompt 引导策略、维护动作、发布前检查都讲清楚。
> 读完后，你应该能够：
> 1. 知道每个模块在做什么。
> 2. 知道为什么会出现某类故障。
> 3. 知道如何最小风险修复。
> 4. 知道发布前到底要验证什么。

---

## 0. 本文档怎么用

### 0.1 推荐阅读顺序

1. 先看第 1 章（项目定位）和第 2 章（目录与边界）。
2. 再看第 3-6 章（消息主链路、Router、Agent、Tool）。
3. 如果你在调 Prompt，重点看第 7-9 章。
4. 如果你在做稳定性，重点看第 10-13 章。
5. 发布前必须执行第 14-16 章清单。

### 0.2 这份文档不承诺什么

- 不承诺第三方站点永远可访问。
- 不承诺 OneBot 上游 100% 稳定。
- 不承诺任何“绝不出问题”。
- 这份文档承诺的是：出现问题后有明确定位与恢复路径。

### 0.3 术语约定

- “主链路”：从收到消息到发送最终回复的完整流程。
- “工具链”：Agent 调用的功能工具集合。
- “Prompt 链路”：模板加载、拼接、注入、运行时约束的整体。
- “回退”：某一步失败后自动切换到备选路径。

## 1. 项目定位与设计目标

### 1.1 项目定位

YuKiKo 是一个偏“执行型”的 QQ 群聊机器人系统，不只是聊天模型壳。
它同时承担：

- 聊天与问答。
- 搜索与网页提取。
- 下载与文件上传。
- 语音/图片/视频多模态处理。
- 群管理与插件扩展。
- WebUI 可视化运维。

### 1.2 设计目标（真实工程目标）

- 可靠：坏了可定位，可恢复，可回滚。
- 可维护：配置分层，插件解耦，文档可执行。
- 可扩展：新工具、新插件可并入，不推翻主干。
- 可发布：支持 Linux 一键部署与远程更新。
- 可观察：日志、状态、会话、agent 轨迹可查看。

### 1.3 设计取舍

- 优先可控，不追“最聪明”。
- 优先明确动作，不追“看起来会说”。
- 优先真实结果，不追“漂亮语言”。
- 优先失败可解释，不追“静默失败”。

## 2. 目录结构与模块边界

### 2.1 根目录关键文件

- `main.py`：启动入口，决定 setup/normal 流程。
- `app.py`：NoneBot 事件接入、消息派发、最终发送。
- `install.sh`：Linux 一键部署。
- `bootstrap.sh`：GitHub 远程直装入口。
- `yukiko`：统一运维命令入口（help/start/stop/update/uninstall）。

### 2.2 core 关键模块

- `core/engine.py`：业务编排核心。
- `core/agent.py`：Agent 多步工具执行。
- `core/agent_tools.py`：工具注册、Schema、Handler。
- `core/router.py`：路由决策。
- `core/thinking.py`：纯回复正文生成。
- `core/webui.py`：WebUI API。
- `core/queue.py`：并发队列与会话调度。
- `core/prompt_loader.py`：Prompt 配置加载与热更新。
- `core/prompt_policy.py`：Prompt 注入策略。
- `core/system_prompts.py`：系统提示词中继。

### 2.3 配置目录

- `config/config.yml`：主配置。
- `config/prompts.yml`：可编辑 Prompt 模板。
- `config/templates/master.template.yml`：默认模板与回填源。
- `plugins/config/*.yml`：插件独立配置。

### 2.4 WebUI 目录

- `webui/src/pages/chat.tsx`：会话页、Thinking 流显示。
- `webui/src/api/client.ts`：API 类型与请求封装。
- `webui/src/styles/globals.css`：全局样式。

### 2.5 文档目录

- `docs/zh-CN/GUIDE.md`：中文使用指南。
- `docs/zh-CN/ARCHITECTURE.md`：简版原理说明。
- `docs/zh-CN/RELEASE_PLAYBOOK.md`：发布运维超详细手册。

## 3. 启动与运行模式

### 3.1 正常模式

1. 读取环境变量。
2. 加载主配置与模板补齐。
3. 初始化插件与工具。
4. 启动 WebUI API。
5. 连接 OneBot 适配器。
6. 进入消息循环。

### 3.2 首次配置模式

触发条件通常是主配置缺失。
系统会优先引导 setup 页，若前端未构建则回退 CLI 向导。

### 3.3 部署模式

- `install.sh` 支持交互与非交互参数。
- 可直接远程：`bash <(curl -fsSL .../bootstrap.sh)`。
- 支持 systemd 注册、启动、状态、日志与卸载。

### 3.4 统一运维命令

`yukiko --help` 应覆盖：

- install
- register/unregister
- start/stop/restart/status
- logs
- update
- uninstall

## 4. 消息主链路（从收消息到回消息）

### 4.1 收消息阶段

1. OneBot 推送事件进入 `app.py`。
2. 抽取消息体、发送者、群号、引用信息、媒体片段。
3. 写入基础日志（含 trace）。
4. 判断 to_me 与群聊策略。

### 4.2 触发守卫阶段

- 依据 `trigger_guard` 策略判断是否进入 AI。
- 非指向消息按置信策略过滤。
- 避免机器人在群里“乱接话”。

### 4.3 队列调度阶段

- 按会话键（如 `group:xxx:user:yyy`）入队。
- 控制并发上限、超时、取消策略。
- 支持单会话串行与跨会话并行。

### 4.4 路由阶段

Router 决定：

- 忽略。
- 走轻量回复。
- 走 Agent 工具链。
- 走特殊能力（如图像/音乐等）。

### 4.5 Agent 阶段

- 构建系统 Prompt + 用户消息。
- 迭代执行工具。
- 处理工具失败与回退。
- 产出 `final_answer`。

### 4.6 发送阶段

- 文本分片发送。
- 附件媒体发送。
- 重试与限流处理。
- 最终写回会话历史。

## 5. Router 与 Self-check 的关系

### 5.1 Router 的价值

- 快速判断是否接入 AI。
- 降低无意义工具调用。
- 提前筛掉明显非目标消息。

### 5.2 Router 的局限

- 仍依赖模型判断，可能误判。
- 对隐式指代、多轮上下文会抖动。

### 5.3 Self-check 的作用

- 本地规则兜底。
- 防止高风险误操作。
- 降低“会说不会做”与“乱接话”。

### 5.4 实际建议

- Router 配置偏保守。
- Self-check 覆盖高风险动作。
- 对关键操作（下载/管理）二次确认条件明确。

## 6. Agent 深度解析（当前核心）

### 6.1 Agent 工作方式

- 每轮输出一个 JSON 工具调用或 final_answer。
- 工具结果回灌后继续下一轮。
- 到达步数上限时走 fallback。

### 6.2 关键防线

- 输出格式硬约束。
- 必填参数补齐。
- 工具白名单与权限分级。
- 失败回退工具。
- 总超时预算。

### 6.3 当前已增强点

- Prompt 中加入工具优先级区块。
- Prompt 中加入本轮可用工具细粒度提示。
- Prompt 中加入执行预算硬约束。
- 下载失败类型（HTML/签名不匹配）可自动回退到资源检索。

### 6.4 常见失败模式

- 模型反复 web_search，步数耗尽。
- 已拿到候选但不执行 smart_download。
- smart_download 首个候选错平台，直接失败。
- final_answer 前没有收敛成可执行结论。

### 6.5 维护重点

- 限制重复搜索。
- 放大工具优先级信号。
- 下载任务平台一致性。
- fallback 信息可用户理解。

## 7. Prompt 链路深度说明

### 7.1 Prompt 来源层级

1. `config/prompts.yml`（用户可编辑）。
2. `master.template.yml`（默认模板）。
3. 代码 fallback（最后兜底）。

### 7.2 `prompt_loader.py` 机制

- 加载 prompts.yml。
- 缺失字段按模板回填。
- 支持热重载。
- 记录 digest 便于确认版本。

### 7.3 `system_prompts.py` 机制

- 提供统一“系统提示词中继”。
- 各模块调用统一入口。
- 降低散落 hardcode。

### 7.4 `prompt_policy.py` 机制

- 全局 prefix/suffix 注入。
- 按 channel 注入（router/agent/thinking/...）。
- 按 tool 注入定制提示。
- 可启 low_iq_mode 强约束。

### 7.5 当前链路优势

- 可配置。
- 可回填。
- 可按场景注入。
- 可热更新。

### 7.6 当前链路风险

- 模板写了但拼接处可能漏接。
- 规则太长导致模型抓不住重点。
- 多段语义冲突时执行漂移。
- 回退文案与工具行为不一致。

### 7.7 Prompt 可维护原则

- 原则少而硬。
- 操作顺序明确。
- 错误后下一动作明确。
- 不要求模型“自己悟”。

### 7.8 Prompt 改造应避免

- 只加形容词，不加动作。
- 只讲愿景，不讲失败处理。
- 只说“不要”，不说“该怎么做”。
- 一段写十几个策略且无优先级。

## 8. Prompt 真正引导性的改造方法

### 8.1 可执行提示的四要素

- 触发条件：什么情况下触发。
- 首选动作：第一步必须做什么。
- 失败分支：失败后改做什么。
- 终止条件：何时结束并回复。

### 8.2 下载任务模板（建议）

- 条件：用户要求“安装包/下载/发文件”。
- 首选：`search_download_resources(query,file_type)`。
- 次选：`smart_download(url,prefer_ext)`。
- 失败：换候选、再试一次。
- 收敛：给可下载候选 + 失败原因。

### 8.3 搜索任务模板（建议）

- 条件：用户要外部事实。
- 首选：`web_search`。
- 补证：`fetch_webpage` 或 `scrape_extract` 一次。
- 收敛：先结论后依据，避免贴原始链接墙。

### 8.4 多媒体任务模板（建议）

- 图片：`analyze_image`。
- 语音：`analyze_voice`。
- 视频链接：`parse_video`。
- 本地视频：`analyze_local_video` 或 `split_video`。

### 8.5 步数预算模板（建议）

- 总步数固定。
- 同类搜索最多两次。
- 第 N-1 步开始强制收敛。
- 禁止“明知失败还重复同工具”。

### 8.6 输出控制模板（建议）

- 如果成功：给结果 + 简短依据。
- 如果部分成功：给可执行下一步。
- 如果失败：给已尝试路径与失败原因。
- 禁止只说“失败了”。

### 8.7 模板改动流程

1. 改 `config/prompts.yml`。
2. 同步 `master.template.yml`。
3. 热重载验证。
4. 跑回归样例。
5. 记录变更说明。

## 9. 下载链路专项（你最关心的点）

### 9.1 目标

“相似需求都能通过爬取和候选筛选，尽量拿到正确下载文件，而不是停在网页或错平台文件。”

### 9.2 现有下载链核心

`smart_download` 当前主流程：

1. 接收原始 URL。
2. 尝试视频解析链（若适用）。
3. 网页提取真实下载候选。
4. 重定向链解析。
5. GitHub release 直链解析。
6. NapCat 下载 + HTTP 兜底。
7. HTML 伪装检测与重试。
8. 文件签名校验。
9. 上传群文件（可选）。

### 9.3 已补强行为（本轮）

- 当 `expected_ext` 与首个下载文件签名不匹配时：
  - 不再立即失败。
  - 自动继续尝试其他候选下载链接。
  - 在匹配成功后继续后续流程。
- 当 `smart_download` 返回 HTML/签名不匹配等错误时：
  - Agent 可自动回退到 `search_download_resources`。

### 9.4 平台一致性规则

- 用户说安卓：优先 `apk`。
- 用户说 Windows/PC：优先 `exe/msi`。
- 用户说 macOS：优先 `dmg`。
- 不跨平台“凑成功”。

### 9.5 高风险误区

- 直接把官网首页当下载链接。
- 明明要 apk，却下到 exe 后直接结束。
- 未告知第三方跳转风险。
- 失败后不提供替代候选。

### 9.6 可观测日志建议

- 记录 candidate_pool 长度。
- 记录每次候选尝试 URL。
- 记录签名校验结果。
- 记录是否触发 third-party 拦截。
- 记录 fallback 工具执行结果。

### 9.7 回归样例（下载）

- 用例 1：`帮我找 XXX 安卓安装包`。
- 期望：优先 `apk`，不返回 `exe`。
- 用例 2：`发我 XXX 电脑版安装包`。
- 期望：优先 `exe/msi`。
- 用例 3：首个候选是 HTML。
- 期望：自动提取并继续重试。
- 用例 4：首个候选扩展名不匹配。
- 期望：继续尝试其他候选，不直接失败。
- 用例 5：必须跨第三方域名。
- 期望：触发确认机制（allow_third_party）。

## 10. WebUI 稳定性重点

### 10.1 chat 页面职责

- 会话列表。
- 历史消息。
- agent state（thinking 流）。
- 日志流展示。

### 10.2 常见 UI 问题

- Thinking 浮层超出视口。
- 非当前会话 thinking 不显示。
- 高频轮询导致闪烁。
- 503 时错误噪音过多。

### 10.3 稳定性建议

- 轮询要有退避策略。
- 离线状态要显式展示。
- thinking 数据做去重。
- 浮层位置按实际宽度夹紧。
- 窗口 resize 触发再夹紧。

### 10.4 可维护样式建议

- 对固定定位组件添加 `max-width` 与 `overflow-wrap`。
- 设定内容区 `max-height + overflow-auto`。
- 保证移动端最小留边。

## 11. 配置体系与热更新

### 11.1 分层原则

- `.env`：密钥与运行环境。
- `config.yml`：主业务策略。
- `prompts.yml`：模型引导策略。
- `plugins/config/*.yml`：插件局部策略。

### 11.2 模板回填价值

- 新版本新增字段自动补齐。
- 老配置不容易因缺键崩溃。
- 升级变更更可控。

### 11.3 热更新原则

- prompt 改动尽量热重载验证。
- 不涉及模型 key 与网络参数时避免重启。
- 关键策略变更后做最小回归。

### 11.4 配置变更纪律

- 一次只改一类参数。
- 每次变更记录目的与风险。
- 失败可快速回滚到上一个配置版本。

## 12. 维护策略（持续运行）

### 12.1 每日例行

- 看 `yukiko status`。
- 看 `yukiko logs --lines 200`。
- 关注 OneBot 在线状态心跳。
- 抽样检查最近会话是否有异常失败。

### 12.2 每周例行

- 检查依赖更新风险。
- 检查插件配置漂移。
- 复核下载类成功率。
- 复核 AI 回复是否偏离风格。

### 12.3 每月例行

- 跑一次全量自检（Python 编译 + WebUI build + pytest）。
- 清理无效策略与过时提示词。
- 复盘本月高频故障并沉淀规则。

### 12.4 故障处理优先级

- P0：服务不可用、消息无法发送。
- P1：关键功能失败（下载/搜索/媒体主功能）。
- P2：体验问题（UI、文案、轻微误判）。
- P3：优化项（性能、可读性、重构）。

---

## 13. 代码维护策略（不激进重构）

### 13.1 原则

- 小步修改。
- 明确边界。
- 先测后提。
- 保留回滚点。

### 13.2 优先维护区

- `core/agent.py`：行为决策主轴。
- `core/agent_tools.py`：工具执行稳定性。
- `core/webui.py`：API 稳定与可降级。
- `webui/src/pages/chat.tsx`：用户感知稳定性。

### 13.3 不建议一次性做的事

- Agent 全量重写。
- Tool registry 一次性拆分成过多文件。
- 在发布窗口做大规模 UI 重构。

### 13.4 推荐分阶段维护

- 阶段 1：故障与误行为修复。
- 阶段 2：日志与可观测性增强。
- 阶段 3：模块解耦与测试完善。

## 14. 自检策略（发布前必做）

### 14.1 最低自检集

- `python -m py_compile`（关键模块）。
- `pytest -q`（如果测试集存在）。
- `npm --prefix webui run build`。

### 14.2 建议自检集

- 启动后 OneBot 连通性检查。
- WebUI 核心 API 冒烟。
- 一条下载请求端到端验证。
- 一条图片分析请求端到端验证。

### 14.3 自检报告必须包含

- 执行命令。
- 成功/失败状态。
- 失败摘要。
- 是否阻断发布。

### 14.4 不合格判定

- 任意构建失败。
- 核心链路不可用。
- 关键配置无法加载。
- 回滚路径不明确。

## 15. 发布策略（你准备发版时）

### 15.1 发布前冻结

- 冻结窗口内只允许修复型提交。
- 变更必须可解释。
- 每个修复要有回归点。

### 15.2 发布步骤

1. 拉取最新主干。
2. 执行自检。
3. 更新文档与变更说明。
4. 打 tag（建议）。
5. 推送。
6. 线上验证。

### 15.3 发布后验证

- WebUI 可打开。
- OneBot 已连接。
- 会话列表与历史可读。
- 发送链路正常。
- 核心工具可调用。

### 15.4 回滚准备

- 保留关键日志时间窗。

## 16. 风险清单（真实会踩）

### 16.1 外部依赖风险

- 插件配置缺字段导致隐性退化。

- 离线状态与错误提示不一致。

## 17. Prompt 维护 Playbook（实操模板）

### 17.1 修改前问自己

- 我在解决哪个具体误行为？
- 这个误行为是否可复现？
- 改动后如何验证它改善了？

### 17.2 修改方式

- 优先改 `prompts.yml`。
- 同步 `master.template.yml`。
- 必要时补代码注入位。

### 17.3 修改后验证

- 旧用例不退化。
- 新用例命中改进目标。
- 日志可证明路径变化。

### 17.4 典型改造示例

- “不要乱搜” 改为：
  - 同类搜索最多两次。
  - 第二次必须带限定词。
  - 超过两次必须收敛回答。

- “下载要正确” 改为：
  - 先识别平台。
  - 先检索候选。
  - 再落地下载。
  - 错包自动换候选。

### 17.5 Prompt 版本化建议

- 每次改动记录“问题 -> 改动 -> 验证”。
- 使用 digest 或注释标识版本。
- 保留过去两版可快速回退。

## 22. Prompt 规则建议清单（可直接复制改造）

### 22.1 规则模板：工具优先

- 用户请求外部事实时，先工具后回答。
- 工具结果不足时最多补一次，不得无限补查。
- 最终答案必须可执行，不得空泛总结。

### 22.2 规则模板：下载任务

- 平台识别优先于链接选择。
- 优先检索候选，不直接下首页。
- 错包或 HTML 自动换候选。
- 输出必须包含失败原因与下一步建议。

### 22.3 规则模板：多轮上下文

- 优先引用当前会话最近有效上下文。
- 出现代词必须先解指代对象。
- 不确定时明确询问，不可脑补。

### 22.4 规则模板：风险控制

- 涉及高风险操作先确认权限。
- 涉及跨域下载先确认是否允许第三方。
- 涉及不可逆动作必须再次确认。

### 22.5 规则模板：输出风格

- 先结论再依据。
- 尽量短句、自然中文。
- 禁止模板腔与工具原样复述。

## 23. 典型故障与处理手册（短版）

### 23.1 症状：WebUI 会话接口 503

排查顺序：

1. 检查 OneBot 是否连接。
2. 检查后端是否在启动中。
3. 检查 `core/webui.py` 对离线态是否降级返回。
4. 检查前端是否对离线态做退避与静默处理。

### 23.2 症状：thinking 不显示

排查顺序：

1. 检查 logs websocket 是否连接。
2. 检查 trace 是否被过滤。
3. 检查是否只显示当前会话。
4. 检查 state 轮询间隔与去重。

### 23.3 症状：找到了下载页但没下载文件

排查顺序：

1. 看 `smart_download` 是否拿到 HTML。
2. 看是否出现签名不匹配（apk vs exe）。
3. 看是否触发第三方拦截。
4. 看是否回退到资源检索。

### 23.4 症状：Agent 达到 max_steps

排查顺序：

1. 看是否重复同类搜索。
2. 看 prompt 是否给出收敛策略。
3. 看工具失败后是否触发 fallback。
4. 看最后 fallback 文案是否可执行。

## 28. 附录 C：维护者心法（经验项）

- 不要追求“永不出错”，要追求“出错可恢复”。
- 不要追求“Prompt 越长越安全”，要追求“规则越明确越稳定”。
- 不要让模型在失败时沉默，必须让它给出下一步行动。
- 不要在发布窗口做大重构。
- 文档是发布的一部分，不是可选项。

## 29. 结语

这份文档不是为了“看起来完整”，而是为了让你和后续维护者在压力场景下仍然能稳定推进。

如果把 YuKiKo 看成一个长期运行系统，那么最重要的并不是单次回答有多华丽，而是：

- 它是否能持续给出可执行结果。
- 它失败时是否能清晰说明。
- 它升级后是否仍可控可回滚。

做到这三点，才是“可以放心发布”的底层标准。
